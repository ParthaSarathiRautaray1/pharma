# Deployment

PharmaCare ships as two deployables from one monorepo:

| App | Artifact | Demo host | Production host |
|---|---|---|---|
| `apps/web` | Static SPA (`dist/`) | Vercel | Nginx static / Vercel |
| `apps/api` | Node server (`dist/server.js`) | Render | Ubuntu VPS + PM2 + Nginx |
| PostgreSQL | Managed / self-hosted | Supabase | PostgreSQL on VPS |

The frontend calls the API cross-origin: the browser hits Vercel for HTML/JS and
the Render/VPS origin for `/api/v1/*`. CORS on the API is what authorizes it, so
the two origins must know each other (`VITE_API_URL` on web, `CORS_ORIGINS` on api).

Config files in the repo:
- [`render.yaml`](../render.yaml) — Render Blueprint for the API
- [`vercel.json`](../vercel.json) — Vercel build + SPA rewrite for the web app
- [`apps/api/ecosystem.config.cjs`](../apps/api/ecosystem.config.cjs) — PM2 for the VPS path

## 1. Database — Supabase (already live)

The demo database is provisioned and seeded. For a fresh environment:

1. Create a Supabase project. Note two connection strings from **Project Settings → Database**:
   - **Transaction pooler** (port `6543`) → `DATABASE_URL`. Append `?pgbouncer=true`.
   - **Session / direct** (port `5432`) → `DIRECT_URL` (migrations only — pgbouncer can't run DDL).
2. Apply the schema and seed demo data:
   ```bash
   cd apps/api
   DATABASE_URL=... DIRECT_URL=... npm run prisma:deploy   # prisma migrate deploy
   DATABASE_URL=... DIRECT_URL=... npm run prisma:seed      # demo tenant + counters
   ```
   > The seed initializes the invoice/purchase number **counters** to the seeded
   > max. Skipping it and hand-loading documents will make the first runtime
   > sale/purchase collide on its number — see PROGRESS.md “Modules 9-13”.
3. **Region matters.** Co-locate the API with the DB. From a distant region the
   multi-step stock transactions (checkout, purchase receive) approach the
   interactive-transaction timeout; `config/prisma.ts` allows 20s of headroom,
   but same-region keeps requests sub-second.
4. **Storage:** create a Storage bucket named `pharmacare` (public or signed as
   you prefer) and set `STORAGE_DRIVER=supabase` in the API env. Render’s disk is
   ephemeral, so `STORAGE_DRIVER=local` there would lose uploaded prescriptions/bills.

## 2. API — Render (demo)

Render auto-deploys from `render.yaml` (a Blueprint).

1. **New → Blueprint** → select this repo → Render reads `render.yaml`.
2. Fill the secrets (marked `sync: false`) in the dashboard:
   - `DATABASE_URL`, `DIRECT_URL` (from Supabase)
   - `APP_URL`, `API_URL`, `CORS_ORIGINS` — set after you know the Vercel domain
     (step 3). `APP_URL`/`CORS_ORIGINS` = the Vercel URL; `API_URL` = the Render URL.
   - `SMTP_*`, `SUPABASE_URL`, `SUPABASE_SERVICE_KEY` (optional but recommended)
   - `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET` are auto-generated by the blueprint.
3. Build/start (already encoded in `render.yaml`):
   - **Build:** `npm ci && npm run build:api && npm run prisma:deploy --workspace=apps/api`
   - **Start:** `npm run start --workspace=apps/api`
   - **Health check:** `GET /health`
4. **Free-tier note:** the instance sleeps after ~15 min idle; the first request
   cold-starts (~30–50s). Fine for a demo. `prisma migrate deploy` runs during
   build and is idempotent. On a paid plan, move it to `preDeployCommand` so it
   runs once per deploy instead of once per build.

Verify: `curl https://<your-api>.onrender.com/health` → `{"success":true,...}`.

## 3. Web — Vercel (demo)

1. **New Project** → import this repo. Vercel reads `vercel.json`:
   - Build: `npm run build:web` · Output: `apps/web/dist` · Install: `npm ci`
   - The catch-all rewrite serves `index.html` for client-side routes (`/app/*`),
     so a hard refresh on a deep link doesn’t 404. Real static assets are served
     from the filesystem first.
2. **Environment variable:** `VITE_API_URL = https://<your-api>.onrender.com`
   (no trailing slash — the axios client appends `/api/v1`). Rebuild after setting it.
3. Copy the resulting Vercel URL back into the API’s `APP_URL` and `CORS_ORIGINS`,
   then redeploy the API so CORS accepts the browser origin.

Verify: open the Vercel URL, log in with a demo account (see PROGRESS.md), and
confirm the dashboard loads data from Render.

## 4. CI/CD

[`.github/workflows/ci.yml`](../.github/workflows/ci.yml) runs on every push/PR to
`main`: install → prisma generate → format check → lint → typecheck → build (with
dummy env). Deploys are handled by the platform GitHub integrations — a green push
to `main` auto-deploys web (Vercel) and api (Render). Keep CI green; a red build
blocks nothing by itself, so treat failures as release blockers by policy.

## 5. Production — self-hosted VPS (Ubuntu + PM2 + Nginx)

For moving off free tiers. Only `DATABASE_URL`/`DIRECT_URL` change in code terms.

### 5.1 Provision
```bash
# Node 20 + build tools (bcrypt is native)
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs build-essential nginx postgresql
sudo npm i -g pm2
```

### 5.2 Database (local Postgres)
```bash
sudo -u postgres createdb pharmacare
sudo -u postgres psql -c "ALTER USER postgres PASSWORD 'strong-password';"
# On a single server, DATABASE_URL and DIRECT_URL point at the same instance.
```

### 5.3 Build + migrate + run the API
```bash
git clone <repo> /srv/pharmacare && cd /srv/pharmacare
cp apps/api/.env.example apps/api/.env   # fill real values (NODE_ENV=production)
npm ci
npm run build:api
npm run prisma:deploy --workspace=apps/api
npm run prisma:seed  --workspace=apps/api   # first deploy only

pm2 start apps/api/ecosystem.config.cjs
pm2 save && pm2 startup     # run the printed command to survive reboots
```

### 5.4 Build the web app
```bash
# Point the SPA at the API origin, then build static files
VITE_API_URL=https://pharmacare.example.com npm run build:web
# Serve apps/web/dist as static files (Nginx block below).
```

### 5.5 Nginx (TLS via certbot, SPA + API proxy on one origin)
Serving web and api on the **same** origin avoids CORS entirely; set
`VITE_API_URL=""` in that case so the SPA uses relative `/api`.
```nginx
server {
  server_name pharmacare.example.com;

  root /srv/pharmacare/apps/web/dist;
  index index.html;

  # SPA: fall back to index.html for client routes
  location / {
    try_files $uri $uri/ /index.html;
  }

  # API reverse proxy
  location /api/ {
    proxy_pass http://127.0.0.1:5000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;   # app trusts proxy (trust proxy = 1)
  }
}
```
```bash
sudo certbot --nginx -d pharmacare.example.com   # Let's Encrypt TLS
```

### 5.6 Backups
```bash
# Nightly pg_dump via cron
0 2 * * * pg_dump -U postgres pharmacare | gzip > /var/backups/pharmacare-$(date +\%F).sql.gz
```

### 5.7 Deploy updates
```bash
cd /srv/pharmacare && git pull
npm ci
npm run build:api && npm run prisma:deploy --workspace=apps/api
pm2 reload pharmacare-api          # zero-downtime reload
VITE_API_URL=... npm run build:web # if the frontend changed
```

## Environment variables

API (`apps/api/.env` / Render). See [`apps/api/.env.example`](../apps/api/.env.example):

| Var | Required | Notes |
|---|:-:|---|
| `NODE_ENV` | ✅ | `production` in deployments |
| `PORT` | ✅ | `5000` (Render injects its own) |
| `APP_URL` | ✅ | Web origin (Vercel/VPS URL) |
| `API_URL` | ✅ | API origin |
| `CORS_ORIGINS` | ✅ | Comma-separated allowed browser origins |
| `DATABASE_URL` | ✅ | Supabase **transaction pooler** (6543, `?pgbouncer=true`) |
| `DIRECT_URL` | ✅ | Supabase **direct** (5432) — migrations |
| `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET` | ✅ | ≥32 chars, distinct |
| `BCRYPT_ROUNDS` | | default 12 |
| `SMTP_*`, `MAIL_FROM` | | unset ⇒ dev-log email transport |
| `STORAGE_DRIVER` | | `supabase` on Render, `local` on VPS |
| `SUPABASE_URL` / `SUPABASE_SERVICE_KEY` / `SUPABASE_BUCKET` | | when `STORAGE_DRIVER=supabase` |
| `RATE_LIMIT_WINDOW_MS` / `RATE_LIMIT_MAX` | | defaults 15m / 300 |
| `LOG_LEVEL` | | default `info` |

Web (Vercel / build-time):

| Var | Notes |
|---|---|
| `VITE_API_URL` | API origin, no trailing slash. Empty ⇒ same-origin `/api` (VPS+Nginx). |

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `CONFLICT … purchaseNumber/invoiceNumber already exists` | counters not seeded | run `npm run prisma:seed`, or set the `counter` rows to the current max |
| Checkout/receive returns `INTERNAL`, ~5s latency | interactive-transaction timeout | co-locate API+DB region; timeout is already raised to 20s in `config/prisma.ts` |
| Random `prepared statement already exists` 500s | pgbouncer + stale prepared statements | ensure `?pgbouncer=true` on `DATABASE_URL`; graceful shutdown disconnects Prisma |
| CORS error in browser console | origin not allowed | add the web origin to `CORS_ORIGINS`, redeploy API |
| Deep-link refresh 404s | missing SPA fallback | `vercel.json` rewrite (Vercel) / `try_files … /index.html` (Nginx) |
| First request after idle is slow | Render free-tier cold start | expected; upgrade plan or keep-warm ping |
