# PharmaCare ERP — System Architecture

> Enterprise Pharmacy ERP + Marketing Website
> Version 1.0 · July 2026

---

## 1. High-Level Overview

A **modular monolith** deployed as two applications sharing one monorepo:

```
┌─────────────────────────────────────────────────────────────────┐
│                          MONOREPO (pharma)                      │
│                                                                 │
│  ┌──────────────────────┐        ┌──────────────────────────┐   │
│  │  apps/web (Frontend) │        │  apps/api (Backend)      │   │
│  │  React 19 + Vite     │  HTTPS │  Node.js + Express + TS  │   │
│  │  Tailwind + shadcn   │ ─────► │  Prisma ORM              │   │
│  │  Marketing + ERP SPA │  JSON  │  JWT Auth (RBAC)         │   │
│  └──────────────────────┘        └───────────┬──────────────┘   │
│                                              │                  │
│                                  ┌───────────▼──────────────┐   │
│                                  │  PostgreSQL              │   │
│                                  │  (Supabase / Render →    │   │
│                                  │   later self-hosted VPS) │   │
│                                  └──────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

**Why a modular monolith (not microservices):**
- A single pharmacy (even multi-branch later) does not need distributed-system complexity.
- One deployable API = free-tier friendly (single Render service), simple to migrate to a VPS with PM2.
- Internal module boundaries (feature folders + service layer) let us extract services later **without rewriting business logic**.

## 2. Application Split

| App | Responsibility | Hosting (Demo) | Hosting (Production) |
|---|---|---|---|
| `apps/web` | Marketing website (public) + ERP SPA (authenticated) | Vercel (free) | Nginx static / Vercel |
| `apps/api` | REST API, auth, business logic, PDF/Excel, email, jobs | Render (free) | Ubuntu VPS + PM2 + Nginx |
| PostgreSQL | Single database, multi-branch ready | Supabase (free) | PostgreSQL on VPS + automated backups |

The marketing site and ERP live in **one React app** with route-level code splitting:
- `/` `/about` `/services` … → public marketing routes (lazy-loaded, SEO meta via `react-helmet-async`-style head management)
- `/app/*` → protected ERP routes (dashboard, inventory, billing, …)
- `/auth/*` → login, forgot password, etc.

This keeps demo hosting to a single free Vercel project. If dedicated SEO/SSR is needed later, the marketing routes can be extracted without touching the ERP.

## 3. Backend Architecture (Layered)

Request flow — every request passes through the same pipeline:

```
Request
  → Global middleware (helmet, cors, compression, rate-limit, requestId, logger)
  → Route (feature router)
  → Auth middleware (JWT verify) → RBAC middleware (role guard)
  → Validator (Zod schema — body/query/params)
  → Controller  (HTTP concerns only: parse input, call service, shape response)
  → Service     (business logic, transactions, cross-module orchestration)
  → Repository / Prisma (data access)
  → Response envelope { success, data, meta } | { success:false, error }
  → Error middleware (AppError → HTTP status, Winston logging, audit trail)
```

**Layer rules (SOLID-aligned):**
- Controllers never touch Prisma directly.
- Services never read `req`/`res`.
- All stock mutations (sale, purchase, return, adjustment) go through a single **StockService** inside a Prisma `$transaction` — this is the invariant that guarantees stock correctness.
- Every write to sensitive entities emits an **AuditLog** row (who, what, before/after, when).

**Cross-cutting modules:**
- `AuthModule` — JWT access (15 min) + rotating refresh tokens (7 d, stored hashed in DB), bcrypt(12) passwords, forgot-password via time-limited token email.
- `NotificationModule` — DB-backed notifications (low stock, near expiry, expired, pending payments) generated by scheduled jobs + real-time events on writes.
- `JobsModule` — `node-cron` in-process jobs (expiry scan, low-stock scan, daily summaries). On free tier the Render instance sleeps, so jobs also run lazily on first request of the day (idempotent).
- `DocumentModule` — invoice/report PDF (pdfkit or puppeteer-free `pdfmake`), Excel (`exceljs`), thermal receipt (ESC/POS-friendly 80mm/58mm PDF layout).
- `MailModule` — Nodemailer with SMTP config via env (Gmail app password for demo → any SMTP in production).
- `WhatsAppModule` — `wa.me` deep-link generation only (no paid API): pre-filled invoice summary, bill link, availability message.

## 4. Frontend Architecture (Feature-Based)

```
State strategy
├── Server state  → TanStack Query (fetching, caching, invalidation, pagination)
├── Client state  → Zustand (auth session, UI prefs, POS cart, theme)
└── Form state    → React Hook Form + Zod (schemas shared shape with API validators)
```

- **Feature folders** (`features/inventory`, `features/billing`, …) own their pages, components, hooks, API calls, and types. Shared primitives live in `components/ui` (shadcn) and `components/common`.
- **Axios instance** with interceptors: attaches access token, transparently refreshes on 401 (single-flight refresh queue), normalizes API errors.
- **Route guards**: `<ProtectedRoute roles={[...]}>` wraps ERP routes; role map mirrors backend RBAC so the UI hides what the API would reject.
- **Design system**: Tailwind tokens (healthcare palette — deep teal/emerald primary, slate neutrals, semantic red/amber/green), dark/light via `class` strategy, Framer Motion for page transitions and micro-interactions, skeleton loaders per feature, empty states, `Ctrl+K` global search, keyboard-driven POS (barcode scan = keyboard wedge input).
- **Performance**: route-level `React.lazy`, TanStack Table with server-side pagination/filter/sort everywhere, memoized chart data, image lazy-loading.

## 6. Multi-Branch Readiness (No Rework Later)

Baked in from day one, invisible until needed:

- Every tenant-owned table carries `pharmacyId` (tenant) and `branchId` (nullable initially).
- All queries are scoped through a request context (`pharmacyId` from JWT) — single-tenant today, multi-tenant/multi-branch tomorrow by populating `branchId` and adding a branch switcher.
- Invoice numbering is per-pharmacy sequence (`INV-{FY}-{seq}`) stored in a counters table with row-level locking — collision-safe across branches.
- Stock is tracked per **batch per branch**, so branch-level inventory needs zero schema change.

## 7. Environment & Deployment Independence

All infrastructure knobs live in env vars — code never references a host:

```
DATABASE_URL, JWT_ACCESS_SECRET, JWT_REFRESH_SECRET,
CORS_ORIGINS, SMTP_HOST/PORT/USER/PASS, APP_URL, API_URL,
UPLOAD_DIR (local disk on VPS / ephemeral+DB-stored on Render), PORT
```

| Concern | Demo (free tier) | Production (VPS) | Code change |
|---|---|---|---|
| API process | Render web service | PM2 cluster behind Nginx | none |
| DB | Supabase pooled connection | Local PostgreSQL + pg_dump cron backups | `DATABASE_URL` |
| SSL | Platform-managed | Let's Encrypt via certbot | none |
| CI/CD | GitHub → auto-deploy (Vercel/Render) | GitHub Actions → SSH deploy script | pipeline only |
| File uploads (prescriptions, invoices) | Supabase Storage (free) | Local disk via Multer | storage adapter via env |

A `StorageAdapter` interface (local | supabase) is the only seam needed — selected by env var.
